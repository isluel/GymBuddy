# GymBuddy - 피트니스 소셜 플랫폼

## 1. 프로젝트 소개
> 운동 사진을 공유하고, 다른 사람의 운동 기록을 보며 동기부여를 받을 수 있는 피트니스 소셜 앱
GymBuddy는 헬스 관련 제품을 만들면서 나온 아이디어에서 시작한 프로젝트입니다.
운동 앱은 기록 기능만 있으면 처음에는 사용할 수 있지만, 시간이 지나면 사용자가 계속 들어올 이유가 약해질 수 있다고 생각했습니다. 그래서 단순히 운동 데이터를 저장하는 앱이 아니라, 사용자가 자신의 운동 사진이나 기록을 공유하고, 다른 사용자를 팔로우하면서 자연스럽게 동기부여를 받을 수 있는 구조를 먼저 만들어보려고 했습니다.
현재 H/W 기기는 아직 제작 중이라 구체적인 데이터 구조나 통신 방식이 확정되지 않았습니다. 그래서 초기 버전에서는 H/W 연동보다는 **소셜 기능을 중심으로 백엔드 구조를 먼저 구현**합니다. 이후 H/W 기기에서 운동 데이터를 받을 수 있게 되면, Android 앱이 해당 데이터를 서버로 전달하고 일별 운동 내역이나 운동 현황, 운동 기록 공유 기능과 연결할 예정입니다.

---
## 2. 현재 구현하려는 기능
초기 버전에서는 피트니스 SNS에 필요한 기본 기능을 구현합니다.
- Google OAuth 로그인
- 사용자 프로필 관리
- 프로필 이미지 등록
- 운동 사진 게시글 작성
- 게시글 이미지 업로드
- 팔로우 / 언팔로우
- 팔로잉 기반 피드 조회
- 좋아요 / 댓글
- 인기 게시글
- 사용자 및 게시글 검색
- 알림 기능

---
## 3. 향후 확장 방향: H/W 운동 데이터 연동
H/W 기기는 현재 제작 중이므로, 초기 버전에서는 구체적인 센서 데이터 구조나 기기 통신 프로토콜을 확정하지 않습니다.
향후에는 사용자의 핸드폰이 H/W 기기와 통신하여 운동 데이터를 수집하고, Android 앱이 해당 데이터를 Backend로 전달하는 구조를 추가할 예정입니다.
전달된 운동 데이터는 다음과 같은 기능에 활용할 수 있습니다.
- 일별 운동 내역 조회
- 주간 운동 현황 조회
- 운동 종류별 기록 관리
- 운동 성과 변화 추이 확인
- 운동 기록 기반 게시글 공유
- 팔로잉 사용자 간 운동 랭킹
- 운동 목표 달성 알림

## 사용 기술

이 프로젝트는 기술을 먼저 정해두고 기능을 맞추기보다는, 구현하려는 기능에서 필요한 문제를 기준으로 기술을 선택했습니다.

### gRPC

Android 앱과 Backend 간 통신은 gRPC를 사용합니다.

게시글 작성, 피드 조회, 팔로우, 좋아요와 같은 기능은 REST API로도 충분히 구현할 수 있습니다. 다만 GymBuddy는 향후 H/W 기기에서 수집한 운동 데이터를 Android 앱을 통해 Backend로 전달하는 기능까지 고려하고 있습니다.

운동 데이터는 운동 세션, 세트, 반복 횟수, 운동 시간, 측정값처럼 일정한 구조를 가진 형태로 전달될 가능성이 있습니다. 이러한 데이터를 다룰 때 Android와 Backend가 동일한 .proto 파일을 기준으로 요청/응답 메시지를 명확하게 정의할 수 있다는 점이 장점이라고 판단했습니다.

또한 이후 운동 데이터를 세션 단위로 묶어 전달하거나, 일정 주기로 여러 측정값을 전송해야 하는 경우 gRPC Streaming 방식으로 확장할 수 있다는 점도 고려했습니다.

따라서 GymBuddy에서는 REST API도 가능한 선택지로 두되, 모바일 앱과 Backend 간의 명확한 API 계약, 향후 iOS 확장 가능성, H/W 운동 데이터 전송 구조로의 확장성을 고려해 gRPC를 선택했습니다.

```text
.proto
   ├── Android Client
   ├── iOS Client
   └── Spring Boot Server
```

### Google OAuth + JWT

로그인은 Android 앱에서 Google OAuth로 진행합니다.
Android에서 Google 로그인을 통해 받은 ID Token을 Backend로 전달하면, Backend는 해당 토큰을 검증한 뒤 서비스에서 사용할 자체 JWT를 발급합니다.
Google 계정으로 사용자를 확인하되, 서비스 내부에서는 자체 사용자 ID와 권한을 기준으로 API를 처리하기 위한 구조입니다.

```text
Android Google Login
   ↓
Google ID Token
   ↓
Backend 검증
   ↓
User 생성 또는 조회
   ↓
서비스 JWT 발급
```

### MinIO

게시글이나 프로필에 사용되는 이미지는 DB에 직접 저장하지 않고 MinIO에 저장합니다.

구현 단순성과 파일 검증의 편의성을 위해 Android 앱이 Backend로 이미지를 전송하고, Backend가 MinIO에 저장하는 방식으로 진행합니다.

```text
Android → Backend → MinIO
```

DB에는 이미지 파일 자체가 아니라 다음과 같은 메타데이터만 저장합니다.

```text
post_image
- id
- post_id
- object_key
- image_url
- content_type
- file_size
- created_at
```

이미지를 DB에 저장하지 않는 이유는 이미지 파일이 많아질수록 DB 용량이 빠르게 증가하고, 백업과 복구가 무거워질 수 있기 때문입니다. 따라서 이미지 원본은 Object Storage에 저장하고, DB는 게시글, 사용자, 팔로우, 댓글과 같은 정형 데이터를 관리하는 역할에 집중하도록 설계했습니다.

### Kafka

GymBuddy에서는 게시글 작성, 좋아요, 댓글, 팔로우와 같은 사용자 행동을 이벤트로 발행하고, 후속 작업을 Kafka Consumer에서 비동기로 처리합니다.  
예를 들어 게시글이 작성되면 인기 게시글 점수 반영, 알림 생성 같은 후속 작업이 필요할 수 있습니다. 이 작업들을 게시글 작성 API 안에서 모두 처리하면 응답 시간이 길어지고, 게시글 작성 로직이 랭킹, 알림 기능과 강하게 결합될 수 있습니다.
따라서 핵심 API는 사용자 요청을 처리하고 이벤트를 발행하는 역할에 집중하고, 랭킹, 알림과 같은 부가 기능은 각각의 Consumer에서 처리하도록 분리했습니다.  

```
PostCreatedEvent 
   ├── Ranking Consumer
   └── Notification Consumer
```

이 구조를 사용하면 새로운 후속 기능이 추가되더라도 기존 게시글 작성 로직을 크게 변경하지 않고 Consumer를 추가하는 방식으로 확장할 수 있습니다.

### Redis

Redis는 주로 빠르게 바뀌는 값이나 랭킹 조회에 사용합니다.

예를 들어 인기 게시글은 좋아요 수, 댓글 수, 조회 수에 따라 계속 점수가 바뀔 수 있습니다. 이런 값을 매번 DB에서 집계하기보다 Redis Sorted Set을 사용해 관리하면 더 단순하고 빠르게 조회할 수 있습니다.

사용 예정 예시는 다음과 같습니다.

```text
ranking:post:daily
ranking:post:weekly
feed:user:{userId}
```
